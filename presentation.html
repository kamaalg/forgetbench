Can your AI memory
system actually forget?

ForgetBench measures whether an LLM memory system can delete a fact so it stops surfacing — without destroying the facts that should stay. One number, four failure modes, runs in 10 lines.

Kamal Gurbanov · companion to the LoCoMo memory-evaluation audit

The gap

Memory systems are built to remember.
Almost none are tested on forgetting.

  • Deletion is the hard, untested skill.Research benchmarks find multi-hop forgetting failing as low as ~7% — but they ship as heavy papers, not as a tool you can run.
  • No memory product ships forgetting tests in CI.Mem0, Zep, Letta — none give you a "did the delete actually work?" signal.
  • "We removed the row" ≠ "it no longer surfaces."Staleness and GDPR / right-to-be-forgotten both need the second guarantee, and nobody measures it.
What it measures

Two rates, scored together so they can't be gamed

🗑️ forget_recall

Of the facts that should be gone, the fraction the system actually removed from retrieval. Delete nothing → 0.0.

🛡️ utility_preservation

Of the facts that should survive, the fraction still retrievable. Delete everything → 0.0.

forget_score = F1( forget_recall , utility_preservation )

High only when the system forgets the right things and keeps the rest — so a "delete everything" system scores ~0.

Four failure axes · 12 cases

The ways real memory systems leak

Direct direct

Delete fact F, probe F directly. The floor competency — does basic deletion even work?

Multi-hop multi_hop

Delete a fact another fact is derived from; its content must not leak back via the derived fact.

Conditional conditional

"Delete X but keep Y." Selective deletion of an obsolete value while the current one survives.

Cross-domain cross_domain

Delete F, then probe an unrelated topic; F must not resurface (no collateral leakage).

LeakyMemory is the cautionary pattern. It distills facts into a per-entity profile at ingest; delete() drops the source document but not the profile — so everything stays answerable (utility 1.0) while every deleted fact leaks back (recall 0.0). That's how extraction/summarization memory tends to fail.

10 lines to benchmark your own system

Implement four methods. Get one score.

# pip install -e .  ·  then:
from forgetbench import run, Document, QueryResult

class MyMemory:
    def ingest(self, documents): ...     # iterable[Document]
    def delete(self, doc_id): ...        # your REAL deletion path
    def query(self, query): ...          # -> QueryResult(retrieved=[...])
    def reset(self): ...                # wipe between cases

report = run(MyMemory())
print(report["forget_score"])

One rule: delete must perform your real deletion — faking it is exactly what the multi-hop and cross-domain axes catch. A live Mem0 adapter ships in the box.

What's in v0.1

Built to be trusted and extended

  • Ungameable score — F1 of forget vs preserve; delete-all scores 0 (tested).
  • Case validator — every contributed case must have a real delete target and both a "forget" and a "preserve" probe, or it's rejected on load.
  • Live Mem0 adapter — handles async extraction by polling until memories are queryable (a blind sleep silently scores everything empty).
  • 15 passing tests · synthetic data only · Apache-2.0 — no licensed data, runs anywhere, $0.
